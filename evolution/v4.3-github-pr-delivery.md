# v4.3 — GitHub 네이티브 PR 전달과 위험 기반 자동 병합

## 관찰된 실패

v4.2는 범위 제한 위생 후보 라우팅과 OKF 시간 의미를 구현했지만 자동 관리 작업의 전달은 로컬 파일·commit에서 끝날 수 있었다. 이 방식은 어떤 실행이 무엇을 바꿨는지 GitHub PR 단위로 검토하기 어렵고, 원격 품질 check와 사람 검토 보류, 안전한 변경의 조건부 병합 영수증도 제공하지 못했다.

사용자는 모든 자동 변경을 PR로 남기고, 사전 승인된 저위험 변경만 검증 뒤 자동 병합하며, 사람 판단이 필요한 변경은 설명이 충분한 PR에서 멈추도록 요구했다. 이 방향은 `COL-27B9ADD786ED`, `SPEC-GH-DELIVERY-001`, 승인된 `RFC-03F4FE85BB44`에 고정했다.

## v4.3의 변화

- Agent를 깨우는 시계는 Codex 예약 작업, 의미 작업은 관리인 Skill, 변경 감사·check·검토·병합 제어는 GitHub로 역할을 분리한다.
- 정확한 `winehouse8/auto_wiki`, `main`, `wiki-auto/`, `squash`만 허용하는 저장소 정책을 둔다.
- 결과를 `중요한 변경 없음|자동 병합 후보|자동 병합 대기|사람 검토 필요|차단`으로 fail-closed 분류한다.
- clean 기준 SHA와 고유 실행 브랜치, 명시적 변경 manifest, base/head/tree SHA, 멱등성 키와 비밀 없는 영수증을 요구한다.
- 전체 게이트 후 고정 base 대비 실측 manifest를 다시 만들고 caller boolean보다 우선하며, 금지 diff는 stage 전에, 실제 token exact scan과 staged 불일치는 commit 전에 차단한다.
- 저위험 변경만 원격 필수 check와 SHA 재확인 뒤 squash auto-merge한다.
- 필수 check 이름을 `전체 저장소 품질 게이트`로 고정하고, 비동기 auto-merge는 merge SHA가 생길 때까지 `자동 병합 대기`로 재조정하되 같은 요청을 중복 전송하지 않는다.
- 도구·Skill·prompt·config·governance·schema·workflow, 의미·신뢰·생명주기·raw와 불확실한 변경은 한국어 draft PR까지만 만들고 Agent가 approve·ready·merge하지 않는다.
- `auth/github_token.yaml`의 classic PAT를 exact repository 검증 뒤 process 환경에만 주입하고 Git·로그·영수증·PR에 노출하지 않는다.
- child Git credential helper는 상속 값을 process 범위에서 초기화하고 대상 helper만 비대화형으로 사용하며, `차단` CLI는 nonzero로 종료한다.
- 비밀 없는 `pull_request` 품질 workflow, 불변 commit SHA로 고정한 외부 action과 제어면 `CODEOWNERS`를 둔다.

## 부트스트랩과 단계적 배포

PR에는 기존 기준 ref가 필요하므로 빈 원격 저장소에 깨끗한 로컬 기준 커밋 `5f1d7f0`만 `main`에 한 번 게시했다. 이 예외는 [GitHub 이슈 #1](https://github.com/winehouse8/auto_wiki/issues/1)에 기록했고 branch protection을 적용했으며 이미 소진됐다. 이후 자동 변경의 `main` 직접 push는 정책 위반이다.

첫 GitHub 통합 PR은 `origin/main`에서 만든 별도의 clean linked worktree에서 `begin` 영수증을 먼저 생성한 뒤 승인된 patch만 적용한다. 아직 자기 PR workflow를 원격 필수 check로 사용할 수 없으므로 로컬 전체 게이트를 거친 뒤 사람이 실제 diff를 검토해 병합한다. 이 PR이 병합된 다음 별도의 무해한 canary PR에서 원격 check와 자동 병합을 전진 검증한다. 첫 통합 PR과 canary가 끝나기 전에는 live adapter 또는 자동 병합 경로가 운영 검증됐다고 표시하지 않는다.

## clean linked worktree 전진 시험에서 찾은 격리 경계

첫 publish 전진 시험의 clean linked worktree에는 Git에서 의도적으로 제외한 `raw/quarantine` payload 54개가 없었다. 기존 검증기는 이를 54개 `quarantine artifact missing` 오류로 처리해 token read·commit·remote 호출 전에 중단했다. 단순히 payload를 public 저장소에 복사하면 보안 격리와 공개 경계가 뒤집히므로, 실패를 `AC-GH-026` 회귀로 보존했다.

보수는 두 프로필을 분리한다. 로컬 보관은 `strict-local-custody`로 원문 누락을 실패시키고, exact public delivery/CI만 `public-clean-clone`으로 canonical admission metadata와 상세 사건 anchor를 검증한다. 공개 프로필은 `quarantine_payload_verified=false`를 남긴다. 또한 `raw/quarantine/**`의 추적 변경은 새 파일도 public draft PR 브랜치에 올라가는 순간 공개되므로 `AC-GH-027`에 따라 token·stage·commit·push·PR 전에 차단한다.

다음 publish에서는 검증 명령이 사건 수를 스스로 늘려 content-addressed 릴리스 archive 이름을 계속 바꾸는 순환도 발견했다. publish 재검증은 중복 사건을 남기지 않는 `--no-log`로 실행하고, 실제 작업의 새 fingerprint 때문에 최초 생성된 archive 한 종류만 canonical digest를 재검증해 최종 manifest에 합친다. 이는 임의의 gate 부작용을 허용하는 예외가 아니며 다른 새 경로는 계속 fail closed한다.

## 호환성과 마이그레이션

canonical 지식 상태, 원문, C/S-level과 OKF bundle schema는 바꾸지 않는다. `config/github-delivery.json`, 실행 영수증과 GitHub PR은 기존 상태 위에 추가되는 전달 제어면이다. Codex 예약 작업이 비활성인 환경에서는 기존 수동 관리 흐름이 유지되며, GitHub 전달 정책이 비활성화되거나 preflight가 실패하면 직접 push로 대체하지 않고 전달 차단으로 끝난다.

기존 여덟 개 통합 릴리스 게이트는 그대로 유지한다. GitHub 전달은 아홉 번째 gate를 이미 제공한다고 주장하지 않고, 기존 릴리스 검사와 전체 단위 테스트 뒤 별도의 delivery preflight와 원격 check를 요구한다.

## 자격증명과 남은 위험

classic PAT는 사람과 Agent를 GitHub에서 같은 `winehouse8` 계정으로 나타내고 저장소보다 넓은 권한을 가질 수 있다. v4.3은 토큰 파일의 종류·소유자·`0600`·ignore·미추적 상태와 process-only 주입을 검사하고, 사람 검토 PR을 adapter 단계에서 병합하지 않음으로써 범위를 제한한다. 짧은 수명의 저장소 한정 GitHub App 또는 전용 machine user로 행위자를 분리하는 작업은 후속 RFC다.

GitHub 장애·rate limit·base drift·동시 실행·잘못된 위험 분류, 의미 공격과 장기 권한 오용은 로컬 fixture로 인증할 수 없다. 일회성 seed와 branch protection 적용만 완료됐고 첫 사람 검토 통합 PR·그 뒤 canary는 아직 남아 있다. 따라서 `production_certified=false`를 유지한다.

## 롤백

Codex 예약 작업과 GitHub 전달을 비활성화하고 열린 자동 PR을 삭제하지 않은 채 close한다. classic PAT를 revoke·rotate하고 adapter·workflow·설정을 제거하는 변경도 사람 검토 PR로 수행한다. 기존 issue·PR·merge·comment와 비밀 없는 영수증은 감사 이력으로 보존한다. 이미 병합된 콘텐츠는 이력을 다시 쓰지 않고 새 revert PR로 되돌린다.
