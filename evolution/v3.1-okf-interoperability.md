# v3.1 — OKF 상호운용성

## 계기

사용자가 Wiki 구축 시 [Open Knowledge Format](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md) 패턴을 따르도록 명시했다.

## 검증한 명세

- 공식 버전: 0.1, 2026-07-11 기준 Draft 표시
- 번들: UTF-8 Markdown 문서로 구성된 계층 디렉터리
- 모든 비예약 문서: 비어 있지 않은 `type`을 포함하며 파싱 가능한 YAML 앞부분 메타데이터
- 예약 파일: `index.md`, `log.md`
- 관계: 표준 Markdown 링크
- 인용: 관례적인 `# 인용` 섹션
- 확장: 임의의 생산자 키를 허용하며 소비자는 알 수 없는 키를 보존해야 함
- 비목표: 고정 분류 체계, 저장·질의 인프라, 도메인 스키마 대체

## 변경 사항

- `wiki/`를 정확한 OKF 번들 경계로 정의했다.
- 모든 개념 문서에 형식이 지정된 앞부분 메타데이터를 추가했다.
- Obsidian 위키 링크를 이식 가능한 Markdown 링크로 바꿨다.
- 루트·하위 디렉터리 색인과 OKF 형식 갱신 로그를 추가했다.
- Living Wiki 확장을 정식 제어면과 연결하는 `okf-profile.md`를 추가했다.
- 결정론적 `okf-validate`를 추가하고 전역 검증기에 포함했다.
- 원문 자료, JSON 원장, 평가 스냅샷, 거버넌스, 실행 도구는 교환 번들 밖에 유지했다.

## v4가 아니라 v3.1인 이유

OKF는 하위 호환 이식성과 더 명확한 번들 계약을 추가하지만, 아직 보정된 사실 정확도나 보안 벤치마크 결과를 개선하지는 않는다. 주 버전 변경을 주장하면 근거를 과장하게 된다.

## 중요한 한계

OKF v0.1은 의도적으로 최소한만 정의한다. 주장 신뢰도, 행위자 신원, 증거 독립성, 출처 신뢰성, 검토, 메모리 오염 방어, 하네스 거버넌스를 정의하지 않는다. 이들은 명시적 생산자 확장과 외부 제어면 기록으로 남으며, 공식 OKF 의미 체계인 것처럼 제시해서는 안 된다.

## 롤백

정식 상태는 `state/`에 남고 OKF 문서는 파생된 읽기용 뷰다. 마이그레이션은 원장에서 번들을 다시 생성할 수 있으며, Git과 콘텐츠 주소 기반 스냅샷은 이전 번들을 보존한다.
