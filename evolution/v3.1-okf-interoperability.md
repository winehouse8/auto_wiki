# v3.1 — OKF interoperability

## Trigger

사용자가 Wiki 구축 시 [Open Knowledge Format](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/main/okf/SPEC.md) 패턴을 따르도록 명시했다.

## Verified specification

- Official version: 0.1, marked Draft as of 2026-07-11
- Bundle: hierarchical directory of UTF-8 Markdown documents
- Every non-reserved document: parseable YAML frontmatter with non-empty `type`
- Reserved files: `index.md`, `log.md`
- Relationships: standard Markdown links
- Citations: conventional `# Citations` section
- Extensions: arbitrary producer keys permitted; consumers should preserve unknown keys
- Non-goals: fixed taxonomy, storage/query infrastructure, replacement of domain schemas

## Change

- Defined `wiki/` as the exact OKF bundle boundary.
- Added typed frontmatter to every concept document.
- Replaced Obsidian wikilinks with portable Markdown links.
- Added root/subdirectory indexes and an OKF-format update log.
- Added `okf-profile.md` mapping Living Wiki extensions to the canonical control plane.
- Added deterministic `okf-validate` and included it in the global validator.
- Kept raw artifacts, JSON ledgers, evaluation snapshots, governance, and executable tooling outside the exchange bundle.

## Why this is v3.1, not v4

OKF adds backward-compatible portability and a clearer bundle contract; it does not yet improve calibrated factual accuracy or security benchmark results. A major version claim would overstate the evidence.

## Critical limit

OKF v0.1 is intentionally minimal. It does not define claim confidence, actor identity, evidence independence, source credibility, review, memory poisoning defense, or harness governance. These remain explicit producer extensions and external control-plane records; they must not be presented as official OKF semantics.

## Rollback

The canonical state remains in `state/`; OKF documents are derived/readable views. A migration can regenerate the bundle from the ledger, and Git/content-addressed snapshots preserve the previous bundle.
