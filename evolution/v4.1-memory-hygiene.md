# v4.1 — 메모리 위생과 통제된 학습

RFC 식별자: `RFC-5D91E03B5BC5`
연구 캠페인: `CMP-01DA85AECBAB`

## 이 릴리스가 필요한 이유

v4는 출처 이력과 릴리스 순환을 닫았지만, 설정된 오래됨 기준, 하류 검색 결과, 비파괴 지식 수명주기를 운영화하지는 않았다. AI Engineer 채널과 독립 ACL 2026 논문을 조사한 결과는 “Agent가 자동으로 학습해야 한다”보다 좁은 결론을 지지했다. 실패와 결과 신호는 유용한 검토 수요이지만 어느 쪽도 인식론적 근거 사실은 아니다.

## 추가형 변경

- `state/memory_feedback.json`은 귀속 정보가 있고 개인정보를 최소화한 `helpful`, `harmful`, `irrelevant`, `unknown` 관측을 저장한다.
- `tools/memory_feedback.py`는 `trust_effect=none`, `automatic_action=false`, 안정된 신원·상태 해시, 귀속된 해결, 비파괴 수명주기 전이를 강제한다.
- `tools/memory_hygiene.py`는 상태를 변경하지 않고 고정 시각의 오래됨, 비활성 지식, 메타데이터 전용 출처, 해결되지 않은 우려 피드백을 보고한다.
- `knowledge-lifecycle`은 `active`, `deprecated`, `superseded`, `invalidated`, `archived` 상태를 이유, 행위자, 이력, 사건 해시, 같은 종류의 대체 항목과 함께 기록한다.
- 기본 어휘 검색은 명시적으로 비활성인 상태와 페이지를 제외한다. `--include-inactive`는 감사용 우회 옵션이지 신뢰 재정의가 아니다.
- `wiki-first`, `fresh-check`, `strict-evidence`는 필수 색인 부트스트랩을 보존하면서 메모리 의존 방식을 사용자가 볼 수 있는 정책으로 드러낸다.
- 릴리스 보고서는 메모리 fixture를 고정하고 고정 시각 위생 게이트를 포함하며, 하네스 버전·릴리스 관련 파일 manifest·보고서 해시를 묶고 RFC 종료 전 승인 후 릴리스 사건을 요구한다.

## 의도적으로 넣지 않은 기능

- C 레벨이나 S 레벨 자동 승격 없음
- 피드백 가중 순위 없음
- 학습에 의한 삭제나 원문 자료 제거 없음
- 그래프·벡터 데이터베이스 기본값 없음
- 고정 fixture가 운영 환경 메모리 품질을 인증한다는 주장 없음

## 롤백 경계

피드백 원장과 활성 수명주기 기본값은 추가형이라 이전 판독기가 무시한다. 비활성 v4.1 주장·출처는 다르다. v3.1 판독기가 이를 조용히 활성 상태로 취급할 수 있다. 따라서 마이그레이션이나 v4.1 호환 롤백 판독기가 이 의미를 보존하지 않는 한, 비활성 기록이 있으면 롤백 예행연습은 실패 폐쇄한다. 롤백은 원문 자료나 추가 전용 사건을 삭제하지 않는다.

## 수용 기준

릴리스에는 구조 검증, OKF 검증, 전체 단위 테스트 모음, 고정된 보정·보안·실행 시간 fixture, 고정된 메모리 피드백·수명주기 fixture, 결정론적 위생 출력, 롤백 예행연습이 필요하다. 통과하더라도 `production_certified=false`를 유지한다.
