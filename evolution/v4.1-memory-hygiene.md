# v4.1 — Memory hygiene and controlled learning

RFC: `RFC-5D91E03B5BC5`  
Research campaign: `CMP-01DA85AECBAB`

## Why this release exists

v4 closed the provenance and release loop but did not operationalize configured staleness, downstream retrieval outcomes, or non-destructive knowledge lifecycle. Research of the AI Engineer channel and independent ACL 2026 papers supported a narrower conclusion than “agents should learn automatically”: failure and outcome signals are useful review demand, but neither is epistemic ground truth.

## Additive changes

- `state/memory_feedback.json` stores attributed, privacy-minimal `helpful|harmful|irrelevant|unknown` observations.
- `tools/memory_feedback.py` enforces `trust_effect=none`, `automatic_action=false`, stable identity/state digests, attributed resolution, and non-destructive lifecycle transitions.
- `tools/memory_hygiene.py` reports fixed-time staleness, inactive knowledge, metadata-only sources, and unresolved concerning feedback without mutation.
- `knowledge-lifecycle` records `active|deprecated|superseded|invalidated|archived` with reason, actor, history, event digest, and same-kind replacement.
- default lexical retrieval excludes explicitly inactive state/pages; `--include-inactive` is an audit escape hatch, not a trust override.
- `wiki-first`, `fresh-check`, and `strict-evidence` expose memory reliance as user-visible policy while preserving the mandatory index bootstrap.
- release reports pin the memory fixture, include a fixed-time hygiene gate, bind harness version, release-relevant file manifest and report digest, and require a post-approval release event before an RFC can close.

## Deliberate non-features

- no automatic C-level or S-level promotion
- no feedback-weighted ranking
- no learned deletion or raw artifact removal
- no graph/vector database default
- no claim that fixed fixtures certify production memory quality

## Rollback boundary

The feedback ledger and active lifecycle defaults are additive and older readers ignore them. An inactive v4.1 claim/source is different: a v3.1 reader could silently treat it as active. The rollback rehearsal therefore fails closed while inactive records exist unless a migration or v4.1-compatible rollback reader preserves those semantics. Rollback never deletes raw artifacts or append-only events.

## Acceptance

Release requires structural validation, OKF validation, the full unit suite, pinned calibration/security/runtime fixtures, the pinned memory-feedback/lifecycle fixture, deterministic hygiene output, and rollback rehearsal. Passing remains `production_certified=false`.
