## 고정한 문제

PR #3에서 `SPEC-GH-DELIVERY-001/v1.2.0` 명세는 병합됐지만 실행기는 여전히 `v1.1.0`만 허용한다. 기존 `begin` 영수증을 둔 채 실행기 상수만 바꾸면 `publish`가 정확한 정책 버전 불일치로 차단되며, 반대로 이전 버전을 포괄 허용하면 검토되지 않은 downgrade·skip·재사용 경로가 생긴다.

따라서 base Git 객체에 이미 존재하는 정확한 `v1.1.0 → v1.2.0` edge만 사용하고, 활성화 PR에서는 설정의 활성 버전과 edge 상태만 바꾸며 검증기·workflow·script를 바꾸지 않는 단일 단계 브리지를 고정한다.

## Red 실행

```bash
python3 -m unittest tests.test_github_delivery_transition tests.test_github_delivery_contract -v
```

결과: 종료 코드 **1**. **19개 테스트 중 실패 15개, 오류 2개, 통과 11개**였고 subtest 때문에 실패 표시는 테스트 수보다 많다.

- `build_policy_transition_proof`와 `validate_policy_transition_activation` API가 아직 없다.
- `idempotency_key`가 target 정책과 base 설정 digest를 받지 않는다.
- `begin` parser가 `--target-policy-version`을 거부한다.
- 검증되지 않은 전환 envelope가 core에 들어오면 token loader가 먼저 호출된다.
- 검토된 exact one-hop edge가 `config/github-delivery.json`에 없다.

fake token loader와 fake transport만 사용했다. 실제 token 파일, GitHub network, branch, commit, push, PR, merge 호출은 0회다.

## Red가 고정한 경계

- 지원되는 전환은 `SPEC-GH-DELIVERY-001/v1.1.0 → SPEC-GH-DELIVERY-001/v1.2.0` 하나뿐이다.
- 권한의 원천은 head/worktree가 아니라 receipt의 `base_sha`에서 읽은 `config/github-delivery.json` Git blob이다.
- begin proof는 raw base 설정의 SHA-256, from/to, base SHA, 승인 협업과 RFC를 함께 고정한다.
- same-version, wildcard, skip, downgrade, consumed edge, 승인·RFC 누락, active 불일치는 token과 외부 부작용 전에 차단한다.
- 활성화는 `config/github-delivery.json` 하나에서 `policy_version`과 `policy_transition.state`만 정확히 바꾸는 경우에만 허용한다.
- 활성화 PR에서 `tools/github_delivery.py`, workflow 또는 다른 파일이 바뀌면 차단한다.
- 전환 멱등성 키는 from/to/base-config digest를 포함하고, 기존 일반 전달 키의 바이트 호환성은 유지한다.
- 전환 PR은 항상 draft 사람 검토이며 auto-merge·approve·ready·merge 요청은 0회다.

이 실패를 약화하거나 삭제하지 않고 승인된 RFC의 최소 구현으로 Green을 만든다.

## 보강 적대 Red

초기 Red 뒤 독립 보안 감사에서 드러난 경계를 각각 실패 fixture로 먼저 고정하고 최소 구현 뒤 Green으로 전환했다. 따라서 아래 결과는 위의 초기 19개 Red 수치에는 포함하지 않는다.

- 중복 JSON authority key와 base digest가 일치하는 head-only edge는 strict parser와 base-bound validator가 없으면 승인됐다.
- 비전환 설정의 JSON `true`를 Python 비교에서 같은 `1`로 바꾼 head가 typed canonical 비교 전에는 승인됐다.
- begin 뒤 origin fetch URL 또는 별도 push URL을 바꾼 fixture는 publish가 gate·token·transport에 도달했다.
- local `git replace`로 같은 base SHA 아래 승인 edge가 든 다른 commit을 대입한 fixture는 base proof를 만들 수 있었다.
- v1.2 base로 이동한 뒤 과거 일반 v1.1 영수증은 원격 기준 검사를 transport까지 미뤄 gate와 token loader에 먼저 도달했다.
- 사람이 review 전환 PR을 병합해 base가 이동한 fixture는 실제 merge SHA를 재조정하지 못하고 무조건 차단됐다.
- CRLF base 설정의 실제 Git blob SHA-256은 `51d7e51c…f348`이지만 text mode로 정규화된 proof는 `6b988b6b…72a9`를 기록했다.
- 기준선 fetch가 검증된 exact HTTPS URL 대신 변경 가능한 별칭 `origin`을 실행 대상으로 사용했다.

각 fixture는 fake transport 또는 로컬 임시 Git 저장소만 사용했고 실제 token 파일, GitHub network, push, PR과 merge 호출은 0회다. 현재 회귀 집합은 exact fetch·push URL과 영수증 결속, `GIT_NO_REPLACE_OBJECTS`, Git blob bytes 보존, JSON 타입 보존, 모든 오래된 영수증의 pre-gate·pre-token 차단, 결속된 사람 검토 PR의 읽기 전용 재조정을 계속 검사한다.
