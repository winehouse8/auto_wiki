## 제품 목표

Codex 예약 작업과 일반 사용자 대화가 하나의 재사용 가능한 Skill을 호출해 위키를 유지한다. 사용자는 자연어로 말하며 Skill 이름, CLI 명령, 행위자 ID, 캠페인 ID 또는 위키 모드를 외울 필요가 없다. Skill은 사용자의 의도를 범위가 제한되고 행위자 귀속이 남는 작업 절차로 바꾸며, 긴 대화에 의존하지 않고 위키를 영속 기억으로 사용한다.

사용자에게 보이는 진입점은 두 가지뿐이다.

1. **예약 사용**: 한 번 일일 예약을 설정한 뒤 결과를 예약 작업 받은 편지함에서 확인한다.
2. **수동 사용**: 평소 말로 점검, 지속 연구, 독립 검증 또는 하네스 개선을 요청한다.

`예약 관리`, `위생 전용` 같은 내부 라우팅 이름은 실행 기록에는 남길 수 있지만 사용자에게 선택 메뉴로 요구하지 않는다.

주요 요청 예:

- “매일 위키 위생을 돌리고 실제로 바뀐 것만 알려줘.”
- “이 분야를 매주 계속 조사해줘.”
- “연구 주제를 하나 추가하거나 잠깐 멈춰줘.”
- “위키 하네스 자체를 더 좋게 만드는 방법을 연구해줘.”

## 사용자 경험 모델

호출 방식과 작업 종류를 서로 다른 축으로 다룬다.

### 호출 방식

- **예약 호출**: 항상 하나의 통합 관리 절차를 사용한다. 위생을 먼저 수행하고, 시점이 된 관심사가 있을 때만 캠페인을 만들며, 도래한 캠페인을 최대 하나 진행하고, 검증 후 중요한 차이만 보고한다.
- **수동 호출**: 사용자의 자연어 의도를 아래 작업 종류 하나로 해석한다. 의도가 여러 작업을 명시적으로 결합하면 안전한 선후관계를 설명하고 필요한 작업만 순서대로 수행한다.

### 작업 종류

- **위생**: 링크·분류·생명주기·중복·고립·노후·렌더링·검증 문제를 점검하고 위험이 낮은 정확한 문제만 보수한다.
- **지속 연구**: 사람이 정한 주제를 등록·변경하거나 출처·시간·독립성 예산과 종료 조건 안에서 증거 작업을 진행한다.
- **독립 검증**: 기존 Wiki 결론을 먼저 답으로 사용하지 않고 최신 1차 자료를 독립 조사한 뒤 현재 Wiki와 비교한다.
- **하네스 개선**: 위키·Skill·하네스 실패를 재현하고 연결 명세와 RFC를 거쳐 가장 작은 지속 변경을 제안한다.

`wiki-first`, `fresh-check`, `strict-evidence`는 작업 종류가 아니라 **Wiki 의존도 정책**이다. 호출 방식이나 작업 종류와 섞어서 사용자에게 하나의 모드 목록으로 보여주지 않는다. 독립 검증 작업은 기본적으로 `fresh-check`를 사용하고, 다른 작업은 사용자가 달리 지정하지 않으면 `wiki-first`를 사용한다.

## 비목표

이 Skill은 운영체제 스케줄러를 만들거나, Codex 샌드박스 정책을 우회하거나, 임의 네트워크 접근을 부여하거나, 자신의 RFC를 승인하거나, 신뢰도를 자동 승격하거나, 원문 증거를 삭제하거나, 운영 환경 인증을 주장하지 않는다. 의미적 유사성을 페이지를 병합하거나 연결해야 한다는 증거로 간주하지 않는다. 자격증명 사용과 외부 공개는 `SPEC-GH-DELIVERY-001`의 정확한 저장소·승인 근거·비밀 안전 계약을 만족한 GitHub 전달에만 한정하며 일반 작업 권한으로 확대하지 않는다.

## 권한

예약 실행은 헌장이 이미 허용한 위험이 낮고 되돌릴 수 있는 작업만 수행할 수 있다. 여기에는 결정론적 평가, 파생 뷰 재생성, 초안 주장, 입수 승인된 출처 보존, 정확한 증거 연결, 열린 질문, 범위가 제한된 캠페인 작업과 좁은 링크·메타데이터 보수가 포함된다. 필요한 승인과 구현 근거가 이미 존재하지 않는 한 정책, 신뢰 게이트, 스키마, 프롬프트, Skill 또는 하네스 변경은 RFC에서 멈춘다.

승인된 GitHub 전달 경계에서는 실제 파일 변경을 PR로 공개할 수 있다. 자동 병합은 헌장이 이미 자동 적용을 허용한 저위험 변경과 모든 필수 검증에만 한정한다. 나머지는 사람 검토 PR에서 멈추며 GitHub에 게시할 수 있다는 권한이 의미·정책 변경을 자동 병합할 권한은 아니다.

## 기능 요구사항

### FR-LWS-001 — 부트스트랩과 의도 라우팅

호출할 때마다 현재 `wiki/index.md`를 읽고 관련 개념·주장·출처 문서를 점진적으로 읽는다. 예약 호출인지 수동 호출인지 먼저 구분하고, 수동 호출이면 사용자의 자연어를 위생·지속 연구·독립 검증·하네스 개선 중 하나로 라우팅한다. 그 다음 `wiki-first`, `fresh-check`, `strict-evidence` 중 적용한 Wiki 의존도 정책을 별도로 기록한다. 사용자가 내부 라우팅 이름이나 식별자를 선택하도록 요구하지 않는다.

### FR-LWS-001A — 예약 호출의 고정 오케스트레이션

예약 호출은 `부트스트랩 → GitHub 실행 시작 → 위생 → 도래한 관심사 생성 → 캠페인 최대 하나 진행 → 검증 → GitHub 전달 → 차이 보고` 순서를 고정한다. 시점이 된 연구가 없으면 연구 단계를 건너뛰고, 위생 결함도 없으면 commit이나 PR 없이 `중요한 변경 없음`을 반환한다. 예약 작업의 생성·정확한 실행 시각·활성 상태는 Codex 예약 작업 제품에서 관리하며 저장소의 내부 모드 선택으로 대체하지 않는다.

### FR-LWS-002 — 위생

상태 확인, 고정 시각 메모리 위생, 린트, 구조 검증과 OKF 검증을 실행한다. 끊어지거나 고립된 링크, 연결 가능한 투영이 있는데 일반 텍스트로 남은 canonical ID, 중복 출처 identity, 오래된 생성 요약, 폴더·타입 차이와 명시적 관계 투영을 확인한다. 정확한 구조 결함은 좁게 보수할 수 있지만 의미적 관련성은 검토 후보로 남긴다.

### FR-LWS-002A — 범위 제한 후보 라우팅

위생은 고정 시각의 결정론적 후보 계획을 만든다. 전체 상태와 문서를 모델 없이 구조적으로 스캔한 뒤 구조 위험, 최근 의미 변경 문서, 검토 기한이 지난 주장과 더 최신인 의존성을 시드로 고른다. 문서 링크·주장·증거·대체·캠페인 관계를 설정된 최대 2-hop과 노드 상한 안에서 확장하고, 선택 이유와 경로를 보존한다. 공유 태그·어휘 유사성은 검토 후보에만 사용하며 링크·병합·신뢰 변경을 자동 수행하지 않는다.

### FR-LWS-002B — 논리 충돌 후보와 점진적 검토

동일하거나 겹치는 주제·범위·시점에서 극성·수치·조건이 다른 주장 쌍을 설정된 상한까지 `검토 필요 후보`로 제시한다. 관리인은 우선순위가 높은 후보만 관련 문서와 evidence locator까지 읽고 `확인된 충돌|범위 차이|노후 후보|거짓 양성|근거 필요`로 설명한다. 후보 생성 자체는 `contradicts` evidence, C/S-level, lifecycle 또는 원문을 변경할 수 없다.

### FR-LWS-002C — 문서 시간과 최신성

OKF `timestamp`는 마지막 의미 있는 내용 변경을 나타낸다. 알려진 경우 `created_at`, 실제 원근거 재확인에는 `last_verified_at`, 검토 주기에는 `freshness`, 생명주기 전이에는 `lifecycle_updated_at` 생산자 확장을 사용한다. 단순 render·confidence 재계산·파일 mtime는 문서를 최신으로 만들지 않는다. 정규 상태와 사건 원장이 권위 기록이며 frontmatter는 후보 필터링을 위한 이동 가능한 투영이다.

### FR-LWS-003 — 반복 연구

시점이 된 관심사를 캠페인으로 만들고 도래한 캠페인을 최대 하나 선택한다. 출처·시간·독립성 예산과 종료 조건을 지키며 확인 검색과 반증 검색을 수행하고, 정확한 locator가 있는 새 출처·주장·증거를 정식 게이트로 연결한다. 시점이 된 작업이 없는 주기는 일을 지어내서는 안 된다.

### FR-LWS-003A — 지역 날짜와 공정한 포트폴리오 선택

일일 실행은 `config/wiki.json`의 IANA 시간대로 지역 날짜를 계산하며 같은 날짜의 포트폴리오 전체에서 설정된 캠페인 상한을 넘지 않는다. 여러 관심사가 도래하면 미배정 우선, 오래된 배정, 오래된 완료, 높은 우선순위, 안정적인 ID 순으로 결정론적으로 선택한다. 완료 상태만 cadence와 질문 회전을 진전시키고 실패·차단·취소·계획 전용 결과는 그날의 공정성 차례만 사용한다.

### FR-LWS-003B — 관심사 연구 brief와 자원 제한 논문화

캠페인은 선택한 관심사의 `research_brief`를 불변 복사해 목표, 제약, 단계별 산출물과 성공 기준을 잃지 않는다. 논문화 트랙은 novelty 조사, 반증 가능한 contribution, 강한 baseline, 재현 가능한 실험 계획, 실행 결과, 논문 초고와 내부 반박 검토를 단계적으로 축적한다. Apple M4·16GB 통합 메모리 제약을 지키며 유료 API·새 자격증명·비용 증가는 별도 사람 승인 없이는 실행하지 않는다.

### FR-LWS-003C — 두 프로젝트 분류와 공유 근거

기본 연구 포트폴리오는 `Wiki 하네스 연구`와 `Agent/Training 논문 연구` 두 프로젝트를 사용자에게 보이는 상위 단위로 제공한다. 프로젝트 레지스트리의 ID는 고유하고 파일명으로 안전한 canonical 형식이며, 각 활성 관심사는 존재하는 프로젝트 하나를 참조한다. 생성된 캠페인은 그 `project_id`를 불변 스냅샷으로 보존하므로 관심사가 나중에 이동해도 과거 분류를 바꾸거나 거부하지 않는다. 프로젝트별 Wiki 뷰는 관심사·캠페인·실행과 연결된 주장·출처를 모아 보여주되 claim/source 정규 원장을 프로젝트별로 복제하거나 공유 근거를 독립 증거로 중복 집계하지 않는다. 페이지 `timestamp`는 표시된 정규 상태의 최신 의미 변경 시각을 투영하며 단순 렌더 시각을 쓰지 않는다. 하루 캠페인 상한과 공정 선택은 두 프로젝트를 합친 포트폴리오 전체에 적용한다.

### FR-LWS-004 — 자연어 관심사 제어

사용자가 연구 주제를 추가·변경·일시 정지·재개·취소하면 행위자 귀속과 자연어 확인을 포함한 정식 관심사·캠페인 표현으로 변환한다. 사용자에게 내부 ID를 요구하지 않는다. 전용 관심사 작성기가 생길 때까지 최소한의 검증된 설정 편집보다 먼저 협업 원장에 방향을 보존한다.

### FR-LWS-005 — 통제된 자가 발전

위키·Skill·하네스 개선을 일반 반복 연구 관심사로 다루되 연구 권한과 수정 권한을 분리한다. 예약 조사는 실패를 재현하고 RFC 초안을 만들 수 있다. 자신을 승인하거나 fixture를 약화하거나 승인되지 않은 정책·하네스 변경을 구현할 수 없다.

### FR-LWS-006 — 결과 계약

중요한 변경만 보고한다. 변경 파일, 새로 생기거나 바뀐 주장·출처 ID와 수준, 보수한 관계, 모순, 미해결 불확실성, 사용 예산, 검증 결과, 승인 차단 요소와 다음 정보 이득이 가장 큰 질문을 포함한다. 실질적인 변화가 없으면 `중요한 변경 없음`이라고 알리고 불필요한 파일 변경을 만들지 않는다.

### FR-LWS-007 — 단계적 사용자 여정

설치와 운영 안내는 `Skill 설치 확인 → 예약 프롬프트 수동 시험 → Codex에서 일일 독립 예약 작업 생성 → 최초 실행 몇 회 검토 → 평소 자연어로 관심사 조정` 순서로 설명한다. 저장소는 Skill과 검증 가능한 예약 프롬프트를 제공하고, 활성 예약 작업의 존재와 다음 실행 시각은 Codex 예약 작업 화면에서 확인하도록 명시한다.

기본 배포는 전용 자동화 checkout을 ChatGPT 데스크톱 앱의 로컬 프로젝트로 선택하고 `Asia/Seoul` 매일 20:00 standalone 작업을 만든다. 각 실행은 dirty 상태를 덮거나 reset하지 않고 깨끗한 이전 실행 브랜치에서 `main`으로 전환한 뒤 `origin/main`을 fast-forward할 수 있을 때만 `begin`을 호출한다. 현재 표면에 예약 생성 도구가 없으면 활성화됐다고 기록하지 않고 `Scheduled` 화면에서 다음 실행 시각을 확인하는 사람 단계로 남긴다.

### FR-LWS-008 — GitHub PR 전달

승인된 `SPEC-GH-DELIVERY-001` 설정이 있으면 모든 실제 추적 파일 변경을 고유 실행 브랜치와 PR로 전달한다. 저위험 변경만 필수 검증과 SHA 일치 뒤 auto-merge하고, 제어면·의미·권한·불확실성 변경은 검토 이유와 방법을 적은 draft PR에서 멈춘다. token은 안전 계약을 통과한 뒤 process 환경으로만 사용하고 값이나 파생 비밀을 출력·저장하지 않는다. 무변경 실행은 빈 PR을 만들지 않는다.

## 실패 동작

- 위키 루트가 없거나 접근할 수 없음: 다른 저장소를 검색하거나 수정하지 않고 중단한다.
- 변경 중인 파일 또는 겹치는 동시 편집이 있음: 덮어쓰지 않고 범위를 줄이거나 충돌을 보고한다.
- 네트워크 또는 승인이 거부됨: 범위가 제한된 로컬 발견을 보존하고 차단된 단계를 보고한다.
- GitHub 대상·token·브랜치·manifest·check 계약이 어긋남: 직접 push나 merge로 우회하지 않고 비밀 없는 전달 차단 영수증을 남긴다.
- 입수·보안 판정이 `allow`가 아님: 후보를 승격하지 않는다.
- 품질 게이트 실패: 주기나 RFC를 완료로 표시하지 않고 실패 근거를 보존한다.
- 영향이 큰 사용자 의도가 모호함: 권한 범위를 추정해 넓히지 말고 방향을 확인한다.

## 롤백

설치된 Skill 링크와 버전 관리되는 `skills/living-wiki-steward/` 디렉터리를 제거한다. 프로젝트 분류를 롤백할 때는 프로젝트 파생 뷰와 관심사 레지스트리를 되돌리되 전역 위키 상태, 원문 자료, 주장, 출처, 캠페인과 사건 이력은 보존한다. 연결된 예약 작업은 Codex 예약 작업 화면에서 별도로 일시 정지하거나 삭제한다. GitHub 전달을 롤백할 때는 열린 자동 PR을 close하고 token을 revoke·rotate하되 기존 PR·merge·comment 이력은 삭제하지 않는다.
