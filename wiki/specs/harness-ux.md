# Living Wiki 하네스 사용자 경험과 전달 계약

상태: `RFC-486C24C969AE`에 따라 승인된 설계 계약. 이 명세는 의도한 제품 동작을 설명한다. 문서에 적힌 동작이라도 수용 테스트와 실행 근거를 통과하기 전에는 실제 제공 기능으로 간주하지 않는다.

연결된 구현 명세:

- [Living Wiki 관리인 Skill 제품 요구사항](living-wiki-steward-skill.md) — 수동 실행 또는 예약 작업에서 호출하는 재사용 가능한 Codex 작업 절차.
- [Living Wiki GitHub PR 전달과 위험 기반 자동 병합](github-delivery.md) — 실제 자동 변경을 PR로 감사하고 저위험 변경만 조건부 병합하는 전달 경계.
- [한국어 문서 정책](korean-documentation-policy.md) — 위키와 하네스의 사람이 읽는 문구에 적용하는 언어·검증 계약.
- [RFC-CB109FAC0820](../governance/rfc-cb109fac0820.md) — 매일 20시 이중 연구 포트폴리오, 공정 선택과 자원 제한 논문화 루프.
- [RFC-5E536A146A81](../governance/rfc-5e536a146a81.md) — 두 연구 프로젝트 분류와 공유 근거 탐색 뷰.

## 제품 목표

사용자는 Codex와 자연어로 협력한다. 명령 이름, 행위자 ID, 주장 ID와 위키 모드는 숙련 사용자를 위한 선택 기능일 뿐 사용자가 외워야 할 어휘가 아니다. Codex는 사용자의 의도를 가장 작고 안전한 읽기·조사·유지보수·제안 작업 절차로 변환하고, 추가 권한을 요청하기 전에 중요한 경계를 설명한다.

주요 사용자 경험의 예:

- “이 주제를 계속 연구해줘. 매주 갱신하고 중요한 변화만 알려줘.”
- “오늘 위키 상태와 낡거나 충돌하는 지식을 정리해줘.”
- “이 답은 기존 위키에 끌리지 않게 새로 검증해줘.”
- “이 하네스가 불편한 점을 조사하고 개선안을 시험해줘.”

Codex는 재현성을 위해 동등한 CLI 명령을 제시할 수 있지만, CLI를 기본 사용자 인터페이스로 강요하지 않는다.

## 사용자 여정과 내부 라우팅

사용자에게 보이는 진입점은 **예약 사용**과 **수동 사용** 두 가지다.

1. 사용자는 관리인 Skill을 설치하고 예약 프롬프트를 한 번 수동 시험한다.
2. Codex 예약 작업 화면에서 이 Wiki 프로젝트를 대상으로 일일 독립 작업을 만들고 최초 몇 차례의 결과를 검토한다.
3. 이후에는 “이 주제를 매주 조사해줘”, “오늘 Wiki를 점검해줘”, “기존 결론과 독립적으로 확인해줘”처럼 자연어로 방향을 조정한다.
4. 활성 예약 여부, 정확한 실행 시각과 최근 결과는 Codex 예약 작업 화면에서 확인한다. 저장소 안의 관심사 주기는 실행된 관리인이 연구 도래 여부를 판단하는 규칙이지 Agent를 깨우는 시계가 아니다.

Skill 내부에서는 호출 방식을 `예약|수동`, 작업 종류를 `위생|지속 연구|독립 검증|하네스 개선`, Wiki 의존도 정책을 `wiki-first|fresh-check|strict-evidence`로 각각 분리한다. 이 값들은 감사와 재현을 위한 내부 표현이며 사용자가 모드 메뉴에서 고르거나 ID를 외워야 하는 제품 인터페이스가 아니다. 예약 호출은 항상 `위생 → 도래한 연구 최대 하나 → 검증 → GitHub 전달 → 중요한 차이 보고`를 수행하는 상위 오케스트레이터다. 추적 파일 변경이 없거나 승인된 GitHub 전달 정책이 활성화되지 않은 경우 전달 단계는 외부 변경을 만들지 않고 그 상태를 보고한다.

## 신뢰 사용자 경험

신뢰도는 유용한 최소 단위에 표시하며 페이지 전체를 하나의 진실 점수로 환원하지 않는다.

- `S0–S4`는 명시한 범위에서 출처의 품질과 출처 추적성을 나타낸다.
- `C0–C4`는 주장 단위의 증거 성숙도를 나타내며 참일 확률이 아니다.
- Agent 합성은 파생 관점이다. 자신이 만든 주장에 대한 독립 외부 증거로 사용할 수 없다.
- 의사결정에 영향을 주는 경우 사실 답변에는 주장 수준, 출처 위치, 최신성, 모순 상태와 주요 한계를 드러낸다.
- 출처 평판, 증거의 함의 관계, 주장의 사실성, 모델 불확실성, 생명주기와 최신성을 서로 다른 차원으로 유지한다.

## 위생 트리거 계약

위생 작업은 멱등적이며 수동 또는 예약 방식으로 요청할 수 있다. 스케줄러는 다음 두 트리거를 모두 평가하고 하나의 실행으로 합친다.

1. `daily`: 해당 지역 달력 날짜에 성공한 위생 실행 영수증이 없으면 하루 한 번 실행한다.
2. `change threshold`: 마지막 성공 위생 영수증 이후 변경된 서로 다른 지식·제어 계층 파일 수가 설정 임계값에 도달하면 실행한다.

첫 배포 기본값은 매일 한 번과 20개 파일 같은 보수적 임계값을 함께 사용하는 것이다. 실제 값은 버전 관리되는 설정에 두고 테스트로 고정한다. 위생 실행은 노후도, 끊어진 참조, 비활성 생명주기 누출, 미해결 모순, 고립된 주장·출처, 중복 독립성 그룹, 렌더링 차이와 검증 상태를 확인한다. 위험이 낮고 되돌릴 수 있는 보수는 수행할 수 있지만, 스스로 신뢰도를 승격하거나 원문 증거를 삭제하거나 정책을 다시 쓰거나 외부에 공개해서는 안 된다.

현재 v4.3 저장소는 결정론적인 위생 평가기, 범위 제한 후보 계획과 Codex 예약 작업이 호출할 관리인 Skill·예약 프롬프트를 제공하지만 상주 데몬을 설치하지 않는다. Agent를 정해진 시각에 깨우는 주체는 Codex 예약 작업이다. 실제 예약 작업의 생성·활성화·정확한 시각과 실행 기록은 Codex 제품의 배포 상태이며 저장소만으로 활성 여부를 증명하지 않는다. 저장소 안의 `daily`·변경 임계값은 실행된 관리인이 도래 여부를 판정하는 의미 규칙이며, 자체적으로 Agent를 깨우는 시계가 아니다. 중복 실행 방지 영수증과 변경 임계값 병합은 여전히 별도 구현 범위다.

## 범위 제한 의미 위생 계약

위생은 전체 문서를 모델 문맥에 넣지 않는다. 먼저 모델 호출이 없는 결정론적 전수 검사를 수행하고, 그 결과에서 제한된 후보만 골라 필요한 문서·주장·출처를 점진적으로 읽는다.

1. **전수 구조 검사**: 모든 정규 상태와 비예약 OKF 문서의 frontmatter, ID, 표준 Markdown 링크, 명시적 관계와 콘텐츠 지문을 검사한다.
2. **후보 시드**: 구조 오류와 고립·비활성 누출·기등록 모순·미해결 유해 피드백은 상한과 무관하게 포함한다. 그 밖에는 `timestamp` 기준 최근 변경 문서, 최신성 기한을 넘긴 주장, 의존성이 문서보다 나중에 바뀐 항목을 각각 설정된 개수까지만 고른다.
3. **그래프 확장**: 문서 링크, `claim_ids`, claim→evidence source, `supersedes`, 캠페인 소속 같은 강한 유형 관계만 기본 최대 2-hop과 전체 노드 상한 안에서 너비 우선으로 확장한다. 공유 태그와 어휘 유사성은 약한 후보 점수이며 관계를 자동 생성하지 않는다.
4. **논리 충돌 후보**: 동일하거나 겹치는 주제·범위·시점을 가진 주장 중 극성, 수치, 조건 또는 명시된 반박 관계가 다른 쌍만 제한된 개수로 제안한다. 결과는 `검토 필요 후보`이며 사실 판정, `contradicts` evidence, C-level, S-level 또는 생명주기를 자동 변경하지 않는다.
5. **점진적 의미 검토**: 관리인 Agent는 우선순위가 높은 후보만 claim→evidence→source locator 순서로 읽고 `확인된 충돌|범위 차이|노후 후보|거짓 양성|근거 필요` 중 하나로 설명한다. 정규 모순이나 evidence로 반영하려면 기존 입수·검토·귀속 절차를 별도로 통과해야 한다.

후보 계획은 고정된 `<NOW>`, 입력 지문, 적용한 개수·hop·노드·쌍 상한, 각 항목의 선택 이유와 시작 시드까지의 경로, 자동 신뢰 효과가 없다는 불변조건을 포함해야 한다. 같은 입력과 시각에서는 바이트가 같은 결과를 만들고 파일을 수정하지 않아야 한다.

## OKF 시간 의미 계약

비예약 문서의 OKF `timestamp`는 공식 v0.1 의미에 따라 **마지막 의미 있는 내용 변경 시각**이다. 단순 렌더, 색인 재생성, confidence 재계산 또는 파일 시스템 mtime만으로 진전시키지 않는다.

- `created_at`: 알려진 최초 생성 시각을 보존하는 Living Wiki 생산자 확장이다. 추정할 수 없으면 거짓 시각을 만들지 않고 생략하거나 마이그레이션 기준임을 별도 기록한다.
- `last_verified_at`: 원근거와 적용 범위를 실제로 다시 확인한 시각이다. 단순 점수 재계산과 구분한다.
- `freshness`: 검토 주기를 선택하는 생산자 확장이다.
- `lifecycle_updated_at`: 생명주기 전이 시각이며 내용 수정·검증 시각과 구분한다.
- `rendered_at`: 필요하면 실행 영수증에 기록하지만 최신성·최근 변경 후보의 기준으로 사용하지 않는다.

정규 상태와 추가 전용 사건 원장이 시간의 권위 기록이며 OKF frontmatter는 이동 가능한 탐색 투영이다. 파생 claim·source 문서는 정규 상태에 알려진 시간 필드를 투영하고, 시간 필드는 시간대가 있는 ISO-8601이어야 한다.

## 반복 연구 계약

반복 연구는 사람이 제시한 주제, 희망 주기, 우선순위와 종료 조건에서 시작한다. Codex는 생략된 값에 안전한 기본값을 채우고 관심사를 기록하며, 실행 시점이 된 경우에만 범위가 제한된 캠페인을 만든다. 각 주기는 확인 검색과 반증 검색, 출처 입수 심사, 정확한 증거 위치, 예산 회계, 합성과 다음 정보 이득 질문을 포함해야 한다.

실행 계층은 영수증과 잠금으로 중첩 실행과 중복 주기를 막아야 한다. “매일 연구”는 사용자가 밀린 실행이나 긴급 실행을 명시적으로 요청하지 않는 한 지역 날짜마다 회계된 주기 한 번 이하를 뜻한다. 사용자는 무엇이 바뀌었는지, 신뢰도 변화, 모순과 아직 모르는 내용을 간결한 차이 보고로 받는다.

현재 기본 연구 포트폴리오는 `Asia/Seoul` 지역 날짜의 매일 20:00에 한 번 깨어나 위생을 먼저 수행하고, 도래한 관심사 중 캠페인 최대 하나만 진행한다. 지역 날짜는 rolling 24시간이 아니라 `config/wiki.json`의 IANA 시간대를 사용한다. 같은 지역 날짜에 예약 호출이 재시도되어도 포트폴리오 전체에서 새 캠페인은 설정 상한을 넘을 수 없다.

여러 관심사가 동시에 도래하면 성공 시계와 배정 시계를 분리한다. 완료된 캠페인만 cadence와 질문 회전을 진전시키며, 실패·차단·취소·계획 전용 실행은 성공으로 세지 않는다. 다만 그날 배정된 관심사는 결과와 무관하게 공정성 차례를 사용한다. 한 번도 배정되지 않은 관심사, 가장 오래 배정되지 않은 관심사, 가장 오래 성공하지 못한 관심사, 높은 명시 우선순위, 안정적인 관심사 ID 순으로 선택해 반복 실패나 정적 ID가 다른 트랙을 영구히 굶기지 않게 한다.

관심사는 질문만이 아니라 `research_brief`의 목표, 제약, 단계별 산출물과 성공 기준을 캠페인에 전달한다. Agent/Training 논문화 트랙은 Apple M4·16GB 통합 메모리에서 재현 가능한 추론 시점 하네스·작은 모델 실험을 우선하고, 전체 사전학습이나 검증되지 않은 대형 fine-tuning을 기본 계획으로 삼지 않는다. ChatGPT·Codex 구독을 API 사용 권한이나 API credit으로 간주하지 않으며, 유료 API·새 자격증명·비용 증가는 별도 사람 승인 전에는 계획에서 멈춘다.

사용자에게 보이는 상위 연구 단위는 `Wiki 하네스 연구`와 `Agent/Training 논문 연구` 두 프로젝트다. `config/interests.json`의 프로젝트 레지스트리가 파일명으로 안전한 canonical `project_id`, 한국어 이름, 목표와 상태를 소유하고 각 관심사는 정확히 한 프로젝트를 참조한다. 새 캠페인은 생성 시점의 `project_id`를 불변 분류 스냅샷으로 보존하며, 이후 관심사가 이동해도 과거 캠페인의 등록된 스냅샷을 바꾸거나 거부하지 않는다. 기존 캠페인은 명시적인 1회 분류를 거친다. 일일 상한은 프로젝트별 한 개가 아니라 두 프로젝트를 합친 포트폴리오 전체 한 개이고 공정 선택은 프로젝트 경계를 넘어 적용한다.

출처와 주장은 프로젝트 소유 파일로 복제하지 않고 전역 근거 그래프에 한 번만 보존한다. 프로젝트별 Wiki 파생 뷰는 `프로젝트 → 관심사 → 캠페인 → 실행`을 보여주고, 캠페인이 연결한 주장·출처를 공유 근거로 링크한다. 같은 주장이나 출처가 두 프로젝트에 기여하면 두 뷰에서 같은 canonical 문서를 가리키며 독립 근거 두 개로 중복 집계하지 않는다.

## GitHub PR 전달 계약

승인된 GitHub 전달 정책이 활성화된 작업 사본에서는 관리인의 실제 추적 파일 변경을 `검증 → 고유 브랜치 → PR → 위험 판정 → 조건부 병합`으로 전달한다. Agent를 깨우는 시계와 의미 연구는 Codex 예약 작업이 담당하고, GitHub는 변경 감사·품질 check·검토·병합 제어면을 담당한다.

- 무변경 실행은 빈 commit이나 PR을 만들지 않는다.
- 파생 뷰·정확한 메타데이터 보수처럼 헌장이 자동 적용을 허용한 변경만 모든 로컬·원격 check 성공 뒤 squash auto-merge할 수 있다.
- 제어면, 의미, 신뢰, 생명주기, raw 변경 또는 불확실한 diff는 상세 한국어 메시지가 있는 draft PR까지만 만들고 Agent가 병합하지 않는다.
- 비밀 노출, raw 덮어쓰기, 사건 원장 재작성과 저장소 identity 불일치는 PR 성공으로 낮추지 않고 차단한다.
- classic PAT는 정확히 승인된 저장소의 delivery adapter에만 process 단위로 주입하며, Git·로그·PR·receipt에 저장하지 않는다.

사용자에게 보이는 최종 상태는 `중요한 변경 없음|자동 병합 후보|사람 검토 필요|차단`이다. 대상은 정확히 `winehouse8/auto_wiki`의 `main`이며, clean `5f1d7f0`을 기준 ref로 게시한 일회성 직접 push 예외는 [GitHub 이슈 #1](https://github.com/winehouse8/auto_wiki/issues/1)에 기록하고 이미 소진했다. 이후 자동 작업은 `main` 직접 push로 PR 경로를 우회할 수 없다.

대상 저장소, 토큰 안전 계약, 멱등성, 동시 실행, 위험 경로와 bootstrap 예외의 상세 기준은 `SPEC-GH-DELIVERY-001`이 소유한다. GitHub Actions의 PR 검증은 secret-free 최소 read permission을 사용하고 privileged workflow에서 신뢰하지 않은 PR head를 실행하지 않는다. classic PAT는 인간과 Agent의 GitHub 행위자를 분리하지 못하고 과권한일 수 있으므로, 저장소 한정 GitHub App 또는 전용 machine user 전환은 후속 RFC로 다룬다.

## 통제된 자가 발전

자가 발전은 독립된 반복 연구 주제이지만 조사 권한과 수정 권한은 다르다. 봇은 방법을 발견하고 실패를 재현하며 RFC 초안을 자율적으로 만들 수 있다. 하네스, 프롬프트, 스키마, 신뢰 정책과 거버넌스 변경에는 승인된 RFC 절차, 고정된 수용 근거, 범위가 제한된 구현과 롤백이 필요하다. 같은 Agent가 제안하고 구현한 것은 독립 검토로 세지 않는다.

자가 발전 주기는 평가기를 몰래 약화하거나 실패 fixture를 삭제하거나 권한을 넓히거나 자신을 승인하거나 로컬 회귀 통과를 운영 환경 인증이라고 주장해서는 안 된다.

## 명세 주도 테스트 개발 전달 계약

모든 하네스 변경은 다음 순서를 따른다.

1. 사용자에게 보이는 동작, 불변 조건, 비목표, 권한, 실패 방식과 수용 기준을 이 명세 또는 연결된 더 좁은 명세에 먼저 반영한다.
2. 하네스 코드, 프롬프트, 스키마, 거버넌스, 신뢰 정책, 예약 또는 권한에 닿는 변경이면 RFC를 만들거나 갱신한다.
3. 실패하는 테스트나 재실행 fixture를 먼저 추가하고 예상 실패를 기록한다: **Red → Green → Refactor**.
4. 고정된 회귀 fixture를 약화하지 않으면서 새 테스트를 통과시키는 가장 작은 변경을 구현한다.
5. 대상 테스트를 실행한 다음 `evaluate`, `render`, 고정 시각 `memory-hygiene`, `lint`, `language-validate`, `validate`, `okf-validate`, 전체 단위 테스트와 통합 릴리스 게이트를 실행한다.
6. RFC를 구현 완료로 전환하기 전에 마이그레이션, 롤백, 남은 위험과 운영 환경 인증의 한계를 기록한다.

긴급 보수는 문서 분량을 줄일 수 있지만, 배포 전 재현 가능한 실패 사례와 변경 후 명세 일치 확인은 생략할 수 없다.

## 수용 시나리오

- 사용자는 `campaign-add`를 몰라도 일반적인 한국어로 새 주제를 말할 수 있고, Codex는 눈에 보이는 주기·예산·종료 조건과 함께 관심사나 캠페인을 기록한다.
- 출처가 풍부한 합성 문단에서도 주장마다 증거가 다르면 같은 페이지 안에서 서로 다른 C-level을 표시한다.
- 일일 트리거와 변경 임계값 트리거가 함께 발생해도 경쟁하는 두 실행이 아니라 위생 실행 영수증 하나만 만든다.
- 실패한 위생 실행은 마지막 성공 표시를 진행시키지 않으며 원문 자료를 삭제할 수 없다.
- 자가 발전 조사로 RFC를 자동 생성할 수 있지만 필요한 사람의 권한 없이 정책 변경을 승인하거나 무인 실행기를 설치할 수 없다.
- 연결 명세와 최초 실패 테스트가 없는 하네스 변경은 릴리스 절차에서 거부한다.
- 사용자는 하나의 관리인 Skill만 보며, 예약 호출은 정해진 오케스트레이션을 사용하고 수동 호출은 자연어 의도를 작업 종류로 라우팅한다.
- 문서는 Skill이 구현됐다는 사실과 실제 Codex 예약 작업이 활성화됐다는 사실을 혼동하지 않는다.
- 자동 변경은 PR 번호와 head SHA로 추적되며, 사람 검토가 필요한 PR은 자동화가 approve·ready·merge하지 않는다.
- 사용자는 프로젝트 색인에서 두 연구 축의 관심사·캠페인·실행·공유 근거를 따로 탐색하고, 같은 주장·출처는 프로젝트마다 복제되지 않는다.

## 현재 구현 경계

현재 구현됨: 주장·출처 신뢰 원장, 결정론적 위생 관찰, 주기 기반 캠페인 생성, 범위가 제한된 계획 영수증, 출처·보안 게이트, RFC, 롤백 인식 릴리스 검사, 매 요청 위키 부트스트랩, Codex 예약 작업이 호출할 관리인 Skill과 일일 예약 프롬프트. 범위 제한 의미 위생과 OKF 시간 의미 계약은 승인된 `RFC-6BD49C04ED49`에 따라 `hygiene-plan`, 최대 2-hop 후보 라우팅, 시간 투영과 수용 회귀검사로 구현됐다. 두 프로젝트 레지스트리와 프로젝트별 파생 탐색 뷰는 `RFC-5E536A146A81`의 로컬 수용 테스트와 릴리스 게이트를 통과했으며, 현재 사람 검토 PR 병합 전이라 운영 기준선에는 아직 반영되지 않았다.

GitHub PR 전달 계약과 로컬 코어는 `RFC-03F4FE85BB44`에 따라 구현 중이다. exact-repository 설정, token 비노출 loader, 위험 분류, fake transport 회귀검사, 비밀 없는 PR workflow, 일회성 clean seed, branch protection과 첫 사람 검토 통합 PR 병합은 완료됐다. 현재는 그 뒤 발견한 v1.2 제어면 결함의 후속 사람 검토 PR, 운영 정책 버전 전환과 live canary가 남아 있다. 따라서 자동 전달 또는 auto-merge가 운영 환경에서 검증됐다고 표시하지 않으며 `production_certified=false`를 유지한다.

저장소 안에서 아직 구현되지 않음: 변경 임계값이 자체적으로 실행을 깨우는 데몬, 네트워크 사용이 가능한 별도 자율 연구 실행기, 위생 작업의 잠금·병합 영수증, 자연어 관심사 전용 작성기. Codex 예약 작업을 실제로 생성·활성화하는 단계도 저장소 변경이 아니라 사용자 배포 단계다. GitHub live 전달은 후속 제어면 PR과 운영 정책 버전 전환을 마친 뒤 별도의 무해한 canary로 전진 검증해야 한다. 이 경계를 넘는 기능이 구현될 때까지 Codex가 자연어 의도를 조정하고 `scripts/research-cycle.sh`는 계획만 수행한다.
